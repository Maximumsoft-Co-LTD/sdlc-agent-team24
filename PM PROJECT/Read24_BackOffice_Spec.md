# Read24 — สเปกระบบหลังบ้าน (Back Office: แอดมิน + สำนักพิมพ์)

| หัวข้อ | รายละเอียด |
|---|---|
| ประเภทเอกสาร | สเปกระบบหลังบ้าน (Back Office) — สำหรับแอดมินและสำนักพิมพ์ |
| ขอบเขต | จัดการหนังสือ + อนุมัติ/เผยแพร่/ระงับ + แดชบอร์ดยอดขาย + ส่วนแบ่งรายได้ + จ่ายเงินสำนักพิมพ์ |
| อ้าง requirement | FR-11 (อัปโหลด/ตั้งราคา), FR-18 (หน้ารายได้), FR-20 (อนุมัติ/ระงับ), FR-17 (ส่วนแบ่ง) |
| อ่านคู่กับ | `Read24_DevSpec_Full.md` (ระบบหลัก/ฐานข้อมูล), `Read24.dc.html` (หน้าตา back office) |
| สถานะ | ฉบับร่าง — รอรีวิว |
| วันที่ | 2026-06-15 |

> เอกสารนี้เจาะเฉพาะ "หลังบ้าน" — ฝั่งร้านค้า/ผู้อ่านอยู่ในสเปกหลัก ฐานข้อมูลใช้ร่วมกัน (อ้างตารางจากสเปกหลัก §4)

---

## 1. ระบบหลังบ้านมีไว้ทำอะไร

> 📌 **อ่านง่ายๆ:** เป็นเครื่องมือให้ทีม Read24 และสำนักพิมพ์ จัดการหนังสือ ดูยอดขาย และคิดส่วนแบ่งรายได้ — ไม่ใช่หน้าที่ผู้อ่านทั่วไปเห็น

หลังบ้านมี 2 มุมมองในระบบเดียว แยกตามบทบาท:
- **มุมแอดมิน (Read24):** อนุมัติ/เผยแพร่/ระงับหนังสือ, ดูภาพรวมยอดขายทั้งระบบ, ดูส่วนแบ่ง, จัดการผู้ใช้/สำนักพิมพ์
- **มุมสำนักพิมพ์:** อัปโหลดหนังสือ+ตั้งราคา, ส่งให้อนุมัติ, ดูยอดขายและรายได้ของตัวเอง (เห็นเฉพาะของตน)

---

## 2. บทบาทและสิทธิ์ (Permissions)

> 📌 **อ่านง่ายๆ:** แอดมินเห็นทั้งระบบ, สำนักพิมพ์เห็นเฉพาะของตัวเอง

| การทำงาน | สำนักพิมพ์ (publisher) | แอดมิน (admin) |
|---|---|---|
| อัปโหลด/แก้หนังสือ (ของตน) | ✅ (FR-11) | ✅ (ทุกเล่ม) |
| ส่งหนังสือให้อนุมัติ | ✅ | — |
| อนุมัติ/เผยแพร่/ตีกลับ/ระงับ | — | ✅ (FR-20) |
| ดูยอดขาย/รายได้ (ของตน) | ✅ (FR-18) | ✅ (ทุกราย) |
| ดูภาพรวมทั้งระบบ | — | ✅ |
| จัดการผู้ใช้/สำนักพิมพ์ | — | ✅ |
| ตั้งค่าระบบ (อัตราส่วนแบ่ง ฯลฯ) | — | ✅ |

- ทุก endpoint หลังบ้านต้องผ่าน guard: token + role; สำนักพิมพ์ถูกกรองด้วย `publisher_id` ของตนเสมอ (กันเห็นข้อมูลคนอื่น)
- บันทึกทุกการกระทำสำคัญลง `audit_logs` (ใคร/ทำอะไร/เมื่อไร) — FR-20

---

## 3. โครงหน้าจอหลังบ้าน (ตาม `Read24.dc.html`)

> 📌 **อ่านง่ายๆ:** แดชบอร์ดสรุปยอด + หน้าจัดการหนังสือ + หน้าส่วนแบ่งรายได้ + (สำหรับสำนักพิมพ์) หน้าอัปโหลดและรายได้

### 3.1 แดชบอร์ด (Dashboard)
แสดงภาพรวม ณ ช่วงเวลาที่เลือก:

| การ์ด/ส่วน | แสดงอะไร | ที่มา |
|---|---|---|
| ยอดขายรายเดือน | กราฟยอดขาย (GMV) แยกรายเดือน + เทียบเดือนก่อน | sum(order_items.unit_price) ที่ order paid |
| ส่วนแบ่งรายได้ | Read24 ได้เท่าไร / สำนักพิมพ์ได้เท่าไร | sum(revenue_splits) |
| แยกตามประเภท | สัดส่วน ซื้อ vs เช่า (จำนวน/มูลค่า) | group by order_items.type |
| ความเคลื่อนไหวล่าสุด | รายการสั่งซื้อ/อนุมัติ/อัปโหลดล่าสุด | orders + audit_logs ล่าสุด |
| จำนวนเล่มเผยแพร่ / รออนุมัติ | นับตามสถานะ | books group by status |

> มุมสำนักพิมพ์: ตัวเลขทุกตัวกรองเฉพาะ `publisher_id` ของตน (FR-18)

### 3.2 จัดการหนังสือ (Books)
- ตารางหนังสือ: ชื่อ, ผู้แต่ง, ราคาขาย/เช่า, ยอดขาย, สถานะ (ฉบับร่าง/รออนุมัติ/เผยแพร่/ระงับ/ตีกลับ)
- การกระทำ: แอดมิน → อนุมัติ/ตีกลับ/ระงับ; สำนักพิมพ์ → แก้ไข/ส่งอนุมัติ
- หน้า "ข้อมูลหนังสือ": ฟอร์ม metadata + ราคา + อัปโหลด EPUB + **ตัวคำนวณส่วนแบ่งสด** (ใส่ราคาแล้วโชว์ว่าสำนักพิมพ์ได้/Read24 ได้เท่าไร)

### 3.3 ส่วนแบ่งรายได้ (Revenue)
- สรุปต่อช่วงเวลา: GMV → หักค่าธรรมเนียม → net → Read24 / สำนักพิมพ์
- แยกตามสำนักพิมพ์ (มุมแอดมิน) หรือเฉพาะของตน (มุมสำนักพิมพ์)
- export CSV ได้

### 3.4 หน้าสำนักพิมพ์ (Publisher portal)
- อัปโหลดหนังสือ + ตั้งราคา + ส่งอนุมัติ (FR-11)
- หน้ารายได้: ยอดขาย, จำนวนเล่ม, ส่วนแบ่งที่ได้รับ (70%) ตามช่วงเวลา (FR-18)

---

## 4. โครงสร้างข้อมูลที่หลังบ้านใช้

> 📌 **อ่านง่ายๆ:** ใช้ตารางเดียวกับระบบหลัก เพิ่มแค่ตารางบันทึกการกระทำของแอดมินและการจ่ายเงินให้สำนักพิมพ์

ใช้ร่วมกับสเปกหลัก: `books`, `orders`, `order_items`, `revenue_splits`, `publishers`, `users`
เพิ่มเฉพาะหลังบ้าน:

**audit_logs (FR-20):** id, actor_user_id FK, action enum('submit','approve','publish','reject','suspend','price_change','payout'), target_type, target_id, note, created_at

**payouts (จ่ายเงินสำนักพิมพ์ — เฟสถัดไป):** id, publisher_id FK, period_start, period_end, amount int, status enum('pending','paid'), paid_at, ref
- ใน Sprint แรกของหลังบ้านอาจยังไม่ทำจ่ายอัตโนมัติ — แค่ "ดูยอดที่ต้องจ่าย" ก่อน (ดู §8 out of scope)

**book_status_history (ทางเลือก):** id, book_id, from_status, to_status, actor, reason, created_at — ติดตามเส้นทางการอนุมัติ

---

## 5. API หลังบ้าน (ขึ้นต้น `/api/v1`)

> 📌 **อ่านง่ายๆ:** แยกเป็นกลุ่ม `/admin/*` (ทีม Read24) และ `/publisher/*` (สำนักพิมพ์เห็นเฉพาะของตน)

### สำนักพิมพ์
```
POST /publisher/books                 { metadata, priceBuy, priceRent, rentDays } → draft   (FR-11)
PATCH /publisher/books/:id            แก้ไขขณะยัง draft/rejected
POST /publisher/books/:id/upload-url  → presigned PUT สำหรับ EPUB ขึ้น S3
POST /publisher/books/:id/submit      → ส่งอนุมัติ (status→pending_review)
GET  /publisher/books?status=         → หนังสือของฉัน + สถานะ + ยอดขาย
GET  /publisher/dashboard?from=&to=   → KPI ของฉัน (ยอดขาย/เล่ม/ส่วนแบ่ง)              (FR-18)
GET  /publisher/revenue?from=&to=     → รายละเอียดส่วนแบ่ง + export CSV
```

### แอดมิน
```
GET  /admin/dashboard?from=&to=       → KPI ทั้งระบบ (GMV, ส่วนแบ่ง, ซื้อ vs เช่า, นับสถานะหนังสือ)
GET  /admin/books?status=&publisher=  → ทุกเล่ม + ฟิลเตอร์
POST /admin/books/:id/publish         → เผยแพร่                                          (FR-20)
POST /admin/books/:id/reject  { reason } → ตีกลับ
POST /admin/books/:id/suspend         → ระงับ (ไม่กระทบคนที่ซื้อแล้ว)
GET  /admin/revenue?from=&to=&publisher= → ส่วนแบ่งทั้งระบบ/ต่อสำนักพิมพ์ + export
GET  /admin/publishers                → รายชื่อสำนักพิมพ์ + ตั้งอัตราส่วนแบ่ง
GET  /admin/audit-logs?cursor=        → บันทึกการกระทำ
```
ทุกการกระทำที่เปลี่ยนสถานะ → เขียน `audit_logs` (FR-20)

## 6. การคำนวณตัวเลขในแดชบอร์ด (นิยาม metric)

> 📌 **อ่านง่ายๆ:** ทุกตัวเลขคิดจากคำสั่งซื้อที่ "จ่ายแล้ว (paid)" เท่านั้น เพื่อให้ตรงกัน

| Metric | สูตร (เฉพาะ order.status='paid') |
|---|---|
| GMV (ยอดขายรวม) | Σ order_items.unit_price |
| ยอดขายรายเดือน | GMV group by เดือนของ paid_at |
| ส่วนแบ่ง Read24 | Σ revenue_splits.platform_cut |
| ส่วนแบ่งสำนักพิมพ์ | Σ revenue_splits.publisher_share |
| ซื้อ vs เช่า | นับ/รวมมูลค่า group by order_items.type |
| จำนวนเล่มขาย/เช่า | count(order_items) |
| ของสำนักพิมพ์ราย X | กรอง revenue_splits.publisher_id = X (FR-18) |

- คิวรีหนักให้ใช้ตัวสำรองอ่าน (read replica) + แคชผลลัพธ์ช่วงเวลาที่ดูบ่อย (เช่น เดือนนี้) 1–5 นาที
- ตัวเลขสำนักพิมพ์ "ต้องตรง" กับผลรวม order ของตน (เงื่อนไขเทสของ FR-18)

## 7. ขั้นตอนอนุมัติหนังสือ (Moderation flow, FR-20)

> 📌 **อ่านง่ายๆ:** สำนักพิมพ์ส่ง → แอดมินตรวจ → เผยแพร่ หรือ ตีกลับ; เล่มมีปัญหาภายหลัง → ระงับ

```
draft ──submit──► pending_review ──approve──► published ──suspend──► suspended
                        │                                  ▲
                        └──reject(reason)──► rejected ──แก้แล้ว submit──┘
```
- แอดมินตรวจตาม checklist (อ้างสเปกหลัก §16): EPUB เปิดได้, มีปก/สารบัญ, ราคาครบ, ไม่มี DRM ภายนอก, สแกนไวรัสผ่าน
- เผยแพร่แล้วเล่มจึงโผล่หน้าร้าน (FR-2); ระงับแล้วถอดจากร้านแต่ **คนที่ซื้อไปแล้วยังอ่านได้** (สิทธิ์ไม่ถูกแตะ)
- ทุก transition เขียน audit_logs + (ถ้าใช้) book_status_history

---

## 8. นอกขอบเขตเฟสแรกของหลังบ้าน
จ่ายเงินสำนักพิมพ์อัตโนมัติ (payout settlement), หักภาษี ณ ที่จ่าย/ใบกำกับ, ระบบ role/permission ละเอียดหลายระดับ, A/B test ดีไซน์ — เฟสแรกทำแค่ "ดูยอดที่ต้องจ่าย" + export ให้การเงินทำมือก่อน

---

## 9. ความปลอดภัย/คุณภาพเฉพาะหลังบ้าน

> 📌 **อ่านง่ายๆ:** กันเห็นข้อมูลข้ามสำนักพิมพ์, บันทึกทุกการกระทำ, ตัวเลขต้องตรงกับเงินจริง

- **กันข้อมูลข้ามราย:** ทุกคิวรีฝั่งสำนักพิมพ์ผูก `publisher_id` จาก token เสมอ — ห้ามรับ publisher_id จาก client (กัน IDOR)
- **Audit ครบ:** publish/reject/suspend/price_change/payout ต้องมี log (ใคร/เมื่อไร/เหตุผล) — FR-20
- **ตัวเลขตรงเงินจริง:** ผลรวมแดชบอร์ดต้องกระทบยอดกับ revenue_splits/orders ได้ (ใช้ในงานเทียบยอดรายวัน, NFR-3/5)
- **สิทธิ์เข้าถึง:** `/admin/*` เฉพาะ admin, `/publisher/*` เฉพาะ publisher; ใช้ MFA สำหรับบัญชีแอดมิน (แนะนำ)
- **Export:** CSV ส่วนแบ่ง/ยอดขาย ต้องจำกัดสิทธิ์และบันทึกว่าใคร export

## 10. ตารางจับคู่ ข้อกำหนด → การทดสอบ (หลังบ้าน)

| FR | สิ่งที่ต้องทดสอบ |
|---|---|
| FR-11 | สำนักพิมพ์อัปโหลด+ตั้งราคา→บันทึก draft; ส่งอนุมัติ→pending_review |
| FR-18 | หน้ารายได้สำนักพิมพ์ตรงกับผลรวม order ของตน; เห็นเฉพาะของตน (ลองเรียกของรายอื่น→403) |
| FR-20 | publish→โผล่ร้าน; reject→กลับไปแก้ได้; suspend→ถอดจากร้านแต่คนซื้อแล้วยังอ่านได้; ทุก action มี audit log |
| FR-17 | แดชบอร์ดส่วนแบ่ง = ผลรวม revenue_splits จริง |
| สิทธิ์ | publisher เรียก `/admin/*`→403; ส่ง publisher_id ปลอม→ถูกเพิกเฉย |

## 11. ลำดับงานหลังบ้าน (ส่งต่อ Dev)
1. Role/guard (publisher/admin) + audit_logs
2. พอร์ทัลสำนักพิมพ์: อัปโหลด+ตั้งราคา+ส่งอนุมัติ (FR-11)
3. แอดมิน: คิวอนุมัติ + publish/reject/suspend (FR-20)
4. แดชบอร์ด + คิวรีรวมยอด (อ่านจาก read replica + แคช) (FR-18)
5. หน้าส่วนแบ่งรายได้ + export CSV (FR-17/18)
6. (เฟสถัดไป) payout

> ทำหลังจากระบบหลัก (ซื้อ/เช่า/จ่าย/ส่วนแบ่ง) พร้อม เพราะหลังบ้านอ่านข้อมูลจากตรงนั้น — ตรงกับ Sprint 3 ในสเปกหลัก §17
