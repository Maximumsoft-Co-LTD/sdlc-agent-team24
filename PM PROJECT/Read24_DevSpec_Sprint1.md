## 1. ขอบเขตงาน Sprint 1

> 📌 **อ่านง่ายๆ:** Sprint แรกทำให้ครบวงจรเดียว — ผู้ใช้ค้นหาหนังสือ → ซื้อหรือเช่า → จ่ายเงิน → อ่านได้เลย

**ทำในรอบนี้:** สมัคร/เข้าสู่ระบบ · ดูรายการหนังสือ + ค้นหา + หน้ารายละเอียด · ซื้อขาด · เช่า 7 วัน (พอครบกำหนดตัดสิทธิ์อ่านอัตโนมัติ) · จ่ายเงินตรงด้วยบัตร/QR PromptPay · อ่านในแอป (ไฟล์ EPUB) · ชั้นหนังสือของฉัน · กันคนที่ไม่ได้ซื้อเปิดอ่าน · บันทึกส่วนแบ่งรายได้ของแต่ละคำสั่งซื้อ

**ยังไม่ทำ (ไว้ Sprint ถัดไป):** กระเป๋าเหรียญ/เติมเหรียญ, ตะกร้าหลายเล่ม, ให้สำนักพิมพ์อัปโหลดเอง + หน้ารายได้, หน้าจัดการของแอดมินแบบเต็ม, รีวิว/ระบบแนะนำ, จ่ายรายเดือน, อ่านออฟไลน์, แอปมือถือ — ช่วง Sprint 1 **แอดมินเป็นคนนำหนังสือเข้าระบบเอง** (ผ่านสคริปต์ภายใน)

## 2. เทคโนโลยีที่ใช้ (ข้อเสนอ — ปรับได้ตอนรีวิว)

> 📌 **อ่านง่ายๆ:** เป็นชุดเครื่องมือมาตรฐานที่ทีมส่วนใหญ่คุ้นเคย ถ้าทีมถนัดอย่างอื่นเปลี่ยนได้ โดยที่ดีไซน์ฐานข้อมูลและ API ยังใช้ได้เหมือนเดิม

| ส่วน | ตัวเลือก | เหตุผลสั้นๆ |
|---|---|---|
| หน้าเว็บ (Frontend) | React + TypeScript (Next.js) แบบ responsive | MVP เป็นเว็บก่อน (PRD A-3) |
| ตัวอ่านหนังสือ | `epub.js` ฝั่งเบราว์เซอร์ | แสดงไฟล์ EPUB ผ่านลิงก์ชั่วคราว |
| หลังบ้าน (Backend) | Node.js + NestJS (TypeScript) แบบ REST | โครงสร้างชัด เทสง่าย |
| ฐานข้อมูล | PostgreSQL 15+ | งานเงิน/สิทธิ์ต้องแม่นยำ เชื่อถือได้ |
| ที่เก็บไฟล์ | S3 (หรือเทียบเท่า) | เก็บไฟล์หนังสือ + ปก แล้วปล่อยลิงก์ชั่วคราว |
| ระบบล็อกอิน | JWT + เข้ารหัสรหัสผ่านด้วย Argon2id | ปลอดภัยตาม NFR-4 |
| ระบบจ่ายเงิน | Omise **หรือ** 2C2P (รองรับบัตร + QR PromptPay) | **ต้องเคาะเจ้าก่อน — PRD Q-1** |
| งานเบื้องหลัง/ตั้งเวลา | Redis + ตัวรันงาน (BullMQ) | ใช้ตัดสิทธิ์เช่าที่หมดอายุ (FR-7) |
| โครงสร้างพื้นฐาน | Docker, CI/CD, เครื่องทดสอบ (staging) | NFR-2 / เกณฑ์คุณภาพ |
| การมอนิเตอร์ | เก็บ log + ติดตามประสิทธิภาพ (APM) | NFR-7 |

## 3. ภาพรวมระบบ

> 📌 **อ่านง่ายๆ:** หน้าเว็บคุยกับหลังบ้าน, หลังบ้านเก็บข้อมูลใน PostgreSQL, ไฟล์หนังสืออยู่บน S3 และส่งให้ผู้ใช้ผ่านลิงก์ที่หมดอายุไว ส่วน "สิทธิ์อ่าน" ถือเป็นตัวตัดสินทุกอย่าง

```
[ หน้าเว็บ (Next.js) ]
      │  HTTPS / ล็อกอินด้วย JWT
      ▼
[ หลังบ้าน NestJS ]
   ├─ ล็อกอิน/สมัคร        (FR-1)
   ├─ รายการหนังสือ        (FR-2,3,4)
   ├─ คำสั่งซื้อ           (FR-5,6,8,17)
   ├─ สิทธิ์อ่าน           (FR-5,6,7,10,12)
   ├─ ตัวอ่าน/เนื้อหา      (FR-9,12)
   └─ ตัวเชื่อมระบบจ่ายเงิน ──► [ เกตเวย์จ่ายเงิน ] ──(แจ้งผลกลับ)──► อัปเดตคำสั่งซื้อ
      │
      ├──► [ PostgreSQL ]  (ผู้ใช้, หนังสือ, คำสั่งซื้อ, สิทธิ์อ่าน, ความคืบหน้าการอ่าน, ส่วนแบ่งรายได้)
      ├──► [ S3 ]          (ไฟล์ EPUB + ปก — เปิดผ่านลิงก์ชั่วคราวเท่านั้น)
      └──► [ Redis + ตัวรันงาน ] (ตัดสิทธิ์เช่าหมดอายุ — FR-7)
```

**หลักคิดสำคัญ:** "สิทธิ์อ่าน (entitlement)" คือแหล่งความจริงเดียวว่าใครอ่านเล่มไหนได้ — ทุกครั้งที่ขอเปิดอ่าน ระบบจะเช็กสิทธิ์สดๆ ทันที (ไม่รอแค่ตัวตั้งเวลา เพื่อกันพลาดตาม PRD R-3)

## 5. ระบบล็อกอินและสิทธิ์การเข้าถึง

> 📌 **อ่านง่ายๆ:** เก็บรหัสผ่านแบบเข้ารหัส, ให้ตั๋วเข้าใช้ (token) อายุสั้นต่ออายุได้, และทุกครั้งที่จะเปิดอ่านต้องเช็กว่ามีสิทธิ์จริง

- **รหัสผ่าน:** เข้ารหัสด้วย Argon2id ห้ามเก็บเป็นข้อความธรรมดา (NFR-4)
- **ตั๋วเข้าใช้:** access token อายุ 15 นาที + refresh token (เก็บใน cookie ปลอดภัย) อายุ 30 วัน
- **ทุกหน้าที่เกี่ยวกับเนื้อหา** ต้องเช็กตั๋ว + เช็กสิทธิ์อ่านก่อนเสมอ (FR-12)
- **หน้าแอดมิน** (`/admin/*`) เข้าได้เฉพาะ role=admin
- **จำกัดอัตรา:** ล็อกอินไม่เกิน 10 ครั้ง/นาที/IP, ขอลิงก์เปิดอ่านไม่เกิน 60 ครั้ง/นาที/คน

## 6. รายการ API (REST, ขึ้นต้น `/api/v1`)

> 📌 **อ่านง่ายๆ:** รวมทุกช่องทางที่หน้าเว็บใช้คุยกับหลังบ้าน — สมัคร/ล็อกอิน, ดูหนังสือ, สั่งซื้อ, รับผลจ่ายเงิน, ชั้นหนังสือ, ขอลิงก์เปิดอ่าน

ทุก error ตอบรูปแบบเดียวกัน: `{ "error": { "code": "...", "message": "..." } }`

### ล็อกอิน/สมัคร
```
POST /auth/register   { email, password, displayName }  → 201 { user, accessToken }
POST /auth/login      { email, password }               → 200 { user, accessToken } (+ ตั้ง refresh cookie)
                       รหัสผิด → 401 INVALID_CREDENTIALS                              (FR-1)
POST /auth/refresh    (cookie)                           → 200 { accessToken }
```

### รายการหนังสือ
```
GET  /books?cursor=&limit=20&category=  → 200 { items:[{id,title,author,coverUrl,priceBuy,priceRent,rentDays,rating}], nextCursor }  (FR-2)
GET  /books/search?q=คำค้น&limit=20      → 200 { items:[...] }  (ไม่เจอ → items:[])                                                    (FR-3)
GET  /books/:id                          → 200 { ...รายละเอียดเล่ม... }   ไม่เจอ/ยังไม่เผยแพร่ → 404 BOOK_NOT_FOUND                       (FR-4)
```
หน้าร้านโชว์เฉพาะ `status='published'` (FR-2)

### สั่งซื้อ / จ่ายเงิน
```
POST /orders          { bookId, type:"buy"|"rent", paymentMethod:"card"|"promptpay" }
                      → 201 { orderId, amount, payment:{ provider, clientSecret|qrPayload } }  (FR-5,6,8)
                      เช่าเล่มที่เช่าไม่ได้ → 400 RENT_NOT_AVAILABLE
GET  /orders/:id       → 200 { id,status,type,amount,book }   (ใช้เช็กสถานะ)
POST /payments/webhook (เกตเวย์ยิงมา, ตรวจลายเซ็น) → อัปเดตคำสั่งซื้อ + สร้างสิทธิ์อ่าน + บันทึกส่วนแบ่ง (ทำพร้อมกันทั้งหมด) (FR-8,17)
```

### ชั้นหนังสือ
```
GET  /me/library  → 200 { owned:[...], renting:[{book,expiresAt,daysLeft}], expired:[...] }  (FR-10)
```

### ตัวอ่าน / เนื้อหา (ต้องมีสิทธิ์อ่านที่ยัง active)
```
GET  /books/:id/content-url  → 200 { url, expiresIn:900 }  (ลิงก์ชั่วคราวอายุ ≤15 นาที, FR-9,12)
                             ไม่มีสิทธิ์/หมดอายุ → 403 NO_ENTITLEMENT  (FR-7,12)
GET  /me/progress/:bookId    → 200 { cfi, percent }
PUT  /me/progress/:bookId    { cfi, percent } → 204                                          (FR-9)
```

### แอดมิน (ช่วง Sprint 1 ใช้นำหนังสือเข้าระบบ)
```
POST /admin/books                  → 201 (สร้างฉบับร่าง)
POST /admin/books/:id/publish      → 200 (เปลี่ยนเป็นเผยแพร่)
POST /admin/books/:id/upload-url   → ได้ลิงก์อัปโหลดไฟล์ EPUB ขึ้น S3
```

## 7. ขั้นตอนการทำงานสำคัญ

> 📌 **อ่านง่ายๆ:** เล่าเป็นสเต็ปว่าตอนซื้อ/เช่า, ตอนอ่าน, และตอนเช่าหมดอายุ ระบบทำอะไรบ้าง

### 7.1 ซื้อ/เช่า + จ่ายเงิน (FR-5,6,8,17)
1. หน้าเว็บเรียก `POST /orders` → หลังบ้านสร้างคำสั่งซื้อสถานะ `pending` แล้วสร้างรายการจ่ายกับเกตเวย์ (บัตร: clientSecret / QR: qrPayload) ส่งกลับให้หน้าเว็บ
2. ผู้ใช้จ่ายเงิน (ยืนยันบัตร หรือสแกน QR)
3. เกตเวย์ยิงผลกลับมาที่ **webhook** → หลังบ้านตรวจลายเซ็น แล้ว **ทำพร้อมกันในครั้งเดียว:** เปลี่ยนคำสั่งซื้อเป็น `paid` → สร้างสิทธิ์อ่าน (ซื้อ = ถาวร / เช่า = วันที่จ่าย + จำนวนวันเช่า) → คำนวณและบันทึกส่วนแบ่งรายได้
4. ถ้าจ่ายไม่สำเร็จ → คำสั่งซื้อเป็น `failed` และ **ไม่สร้างสิทธิ์อ่าน** (FR-8)
5. หน้าเว็บเช็กสถานะ `GET /orders/:id` จนเป็น `paid` แล้วปลดล็อกให้อ่าน

> กันยิงซ้ำ: webhook ต้องรองรับการยิงซ้ำได้โดยไม่สร้างซ้ำ (ใช้ `payment_ref` เป็นกุญแจกันซ้ำ)

### 7.2 เปิดอ่านหนังสือ (FR-9,12)
1. เรียก `GET /books/:id/content-url` → ระบบเช็กว่ามีสิทธิ์อ่านที่ยัง active (ถ้าเช่า ต้องยังไม่หมดอายุ)
2. มีสิทธิ์ → ได้ลิงก์ชั่วคราวอายุ ≤15 นาที / ไม่มีสิทธิ์ → 403 NO_ENTITLEMENT
3. ตัวอ่าน `epub.js` โหลดไฟล์มาแสดง และบันทึกความคืบหน้าทุกครั้งที่เปลี่ยนหน้า/ปิด

### 7.3 การเช่าหมดอายุ (FR-7)
- **ตัวรันงาน (ทุก 15 นาที):** เปลี่ยนสิทธิ์เช่าที่เลยกำหนดให้เป็น `expired` → รับประกันตัดสิทธิ์ภายใน 1 ชั่วโมง
- **เช็กสดตอนเปิดอ่าน (กันพลาด, R-3):** ถ้าเป็นการเช่าและเลยกำหนดแล้ว ตอบ 403 ทันที แม้ตัวรันงานยังไม่ทำงาน
- หน้าจอ: เจอ 403 → เด้งหน้าต่าง "เช่าอีกครั้ง / ซื้อขาด" (ดูตัวอย่างใน mockup)

## 8. การป้องกันเนื้อหา (FR-12, NFR-4)

> 📌 **อ่านง่ายๆ:** ไฟล์หนังสือไม่เปิดสาธารณะ ต้องขอลิงก์ชั่วคราวที่หมดอายุไว และให้เฉพาะคนที่มีสิทธิ์

- ไฟล์ EPUB เก็บในที่เก็บแบบส่วนตัว ไม่มีลิงก์สาธารณะ
- เปิดได้ผ่าน **ลิงก์ชั่วคราวอายุ ≤15 นาที** ที่ออกให้เฉพาะคนมีสิทธิ์อ่านเท่านั้น
- ไม่มีปุ่มดาวน์โหลดทั้งเล่ม (MVP ไม่มีอ่านออฟไลน์)
- ระบบกันก๊อปแบบเต็ม (DRM) เป็นงานอนาคต (PRD R-2) — รอบนี้ใช้ลิงก์ชั่วคราว + ไม่มีดาวน์โหลด เป็นมาตรการขั้นต่ำ

---

## 9. การคำนวณส่วนแบ่งรายได้ (FR-17, PRD §9)

> 📌 **อ่านง่ายๆ:** พอจ่ายสำเร็จ ระบบหักค่าธรรมเนียมก่อน แล้วแบ่ง Read24 ~30% / สำนักพิมพ์ ~70%

```
gross           = ยอดที่ผู้ใช้จ่าย
gateway_fee     = round(gross × อัตราค่าธรรมเนียม)        # ตั้งค่าได้ ~0.03
net             = gross − gateway_fee
platform_cut    = round(net × (1 − ส่วนแบ่งสำนักพิมพ์))   # ~0.30
publisher_share = net − platform_cut                     # ใช้ส่วนที่เหลือ กันปัดเศษเพี้ยน
```
บันทึกลงตาราง `revenue_splits` พร้อมคำสั่งซื้อ และต้องเช็กว่า `platform_cut + publisher_share == net`

## 10. รหัส Error มาตรฐาน

| HTTP | code | เกิดเมื่อ |
|---|---|---|
| 400 | RENT_NOT_AVAILABLE / VALIDATION_ERROR | ข้อมูลส่งมาไม่ถูก |
| 401 | INVALID_CREDENTIALS / TOKEN_EXPIRED | ปัญหาล็อกอิน |
| 403 | NO_ENTITLEMENT / FORBIDDEN | ไม่มีสิทธิ์อ่าน/ไม่ใช่แอดมิน (FR-12) |
| 404 | BOOK_NOT_FOUND | ไม่มีเล่ม/ยังไม่เผยแพร่ |
| 409 | DUPLICATE_ENTITLEMENT | ซื้อซ้ำเล่มที่เป็นเจ้าของแล้ว |
| 500 | INTERNAL | อื่นๆ |

## 11. เกณฑ์คุณภาพ/ประสิทธิภาพ (อ้าง PRD §12)

> 📌 **อ่านง่ายๆ:** เปิดเร็ว จ่ายแล้วได้สิทธิ์ครบ 100% บั๊กร้ายแรงต้องเป็นศูนย์

- ความเร็ว (p95): หน้ารายการ/รายละเอียด ≤ 2.5 วิ, ขอลิงก์อ่าน ≤ 1 วิ, เปิดเล่มแรก ≤ 4 วิ (NFR-1)
- จ่ายแล้วได้สิทธิ์ = 100% (มีงานเทียบยอด orders ที่ paid กับ entitlements ทุกวัน, NFR-3/5)
- บั๊กร้ายแรง (Critical) = 0, เทสวงจรหลักผ่าน 100%, ความครอบคลุมเทสของโมดูลเงิน ≥ 80% (PRD §12)
- ทุกข้อมูลลับเก็บใน secret manager, ทุกช่องทางเป็น HTTPS (NFR-4)

## 16. รองรับคนได้เท่าไรตอนเปิดตัว

> 📌 **อ่านง่ายๆ:** ออกแบบให้คนใช้พร้อมกันช่วงพีคราว 500 คน และเผื่อขยายถึง 1,000 คน โดยให้ไฟล์หนังสือโหลดผ่าน CDN ไม่หนักหลังบ้าน

### 16.1 ตัวเลขที่ตั้งไว้ (ปรับได้ตอนรีวิว)

| ตัวเลข | เป้าเปิดตัว (3 เดือนแรก) | ออกแบบให้รองรับเผื่อ |
|---|---|---|
| ผู้ใช้สมัครสะสม | 5,000–20,000 (ตั้งฐาน 10,000) | 50,000 |
| ผู้ใช้ต่อวัน (DAU) | ~1,000 | 5,000 |
| **ใช้พร้อมกันช่วงพีค** | **~500 คน** | **1,000 คน** |
| สัดส่วน อ่าน : เขียนข้อมูล | ~95 : 5 | — |
| คำขอช่วงพีค | ~150 ครั้ง/วินาที | **ออกแบบรองรับ 500 ครั้ง/วินาที** |

> ตัวเลขนี้ใช้แทนค่า "~50 คนพร้อมกัน" ใน PRD ที่เป็นแค่ตอนทดสอบ/เดโม

### 16.2 เป้าความเร็วตอนมีคนใช้ 500 คนพร้อมกัน
หน้ารายการ/รายละเอียด ≤ 2.5 วิ · ขอลิงก์อ่าน ≤ 1 วิ · เปิดเล่มแรก ≤ 4 วิ · อัตราพลาด < 0.5% · ออนไลน์ ≥ 99% (NFR-2)

### 16.3 ขนาดเครื่องเริ่มต้น (รองรับ ~1,000 คนก่อนขยายอัตโนมัติ)

| ส่วน | เริ่มต้น | ขยายยังไง |
|---|---|---|
| หลังบ้าน (API) | 2–3 เครื่อง (2 vCPU/4GB) หลังตัวกระจายโหลด | เพิ่มเครื่องอัตโนมัติเมื่อ CPU เกิน 65% |
| ฐานข้อมูล | ตัวหลัก (4 vCPU/16GB) + ตัวสำรองอ่าน 1 ตัว | งานอ่านรายการไปตัวสำรอง + คุมการเชื่อมต่อด้วย PgBouncer |
| Redis | 1 เครื่อง (2GB) | งานเบื้องหลัง + จำกัดอัตรา + แคช |
| ที่เก็บไฟล์ + CDN | S3 + CDN | **ไฟล์หนังสือ/ปกโหลดผ่าน CDN** ด้วยลิงก์ชั่วคราว → ลดภาระหลังบ้าน |
| ตัวรันงาน | 1 เครื่อง | แยกจาก API |

### 16.4 หลักสำคัญในการรับโหลด
- **หลังบ้านไม่เก็บสถานะในตัวเอง** → เพิ่มเครื่องได้ทันทีเวลาคนเยอะ
- **ไฟล์หนังสือไม่วิ่งผ่านหลังบ้าน** แต่โหลดตรงจาก CDN ด้วยลิงก์ชั่วคราว หลังบ้านแค่ออกลิงก์ (งานเบา) จึงรับคนเยอะได้
- **แคชหน้ารายการหนังสือ** (คนดูเยอะ เปลี่ยนช้า) เก็บไว้ 1–5 นาที ลดภาระฐานข้อมูล
- ฐานข้อมูลเขียนหนักเฉพาะตอนจ่ายเงิน ซึ่งมีไม่มาก
- **ทดสอบโหลดก่อนเปิดตัว** จำลอง 500–1,000 คนพร้อมกัน ต้องผ่านเป้าข้อ 16.2 (เป็นเงื่อนไขใน PRD §12)
- เปิดตัวแบบ **ค่อยๆ เปิดทีละกลุ่ม** เพื่อคุมโหลดช่วงแรก

## 17. การรับหนังสือเข้าระบบตอนเปิดตัว

> 📌 **อ่านง่ายๆ:** ตั้งเป้ามีหนังสือพอให้ร้านไม่โล่ง (~500 เล่ม) ช่วงแรกแอดมินเป็นคนนำเข้าให้เอง พร้อมตรวจไฟล์ก่อนเผยแพร่

### 17.1 เป้าจำนวนเล่มวันเปิดตัว
อย่างน้อย **300–1,000 เล่ม** (ตั้งเป้า ~500) และต้องมีกลุ่ม **เล่มที่เช่าได้ + เล่มที่ลงกับเราเจ้าเดียว** ตามจุดขายใน PRD §11 — เน้นดีลสำนักพิมพ์รายเล็ก/นักเขียนอิสระก่อน

### 17.2 ช่องทางนำเข้า (Sprint 1)
รอบนี้ยังไม่เปิดให้สำนักพิมพ์อัปโหลดเอง → ใช้ **แอดมินนำเข้า** 2 ทาง:
1. **นำเข้าทีละมากๆ** — แอดมินอัปโหลดไฟล์ข้อมูลหนังสือ (CSV/Sheet) + ไฟล์ EPUB ขึ้น S3
2. **ทีละเล่ม** — ผ่านฟอร์มภายในของแอดมิน

(ให้สำนักพิมพ์อัปโหลดเอง = Sprint 3, FR-11)

### 17.3 ข้อมูลขั้นต่ำต่อเล่ม
ชื่อเรื่อง, ผู้แต่ง, สำนักพิมพ์, หมวด, เรื่องย่อ, ปก, ไฟล์ EPUB, ราคาซื้อ, ราคาเช่า (หรือว่าง), จำนวนวันเช่า, ข้อมูลสัญญา/สิทธิ์, ขายอย่างเดียวหรือขาย+เช่า, พื้นที่ขาย (ไทย)

### 17.4 ตรวจไฟล์ก่อนรับเข้า
- เป็นไฟล์ EPUB ที่ถูกต้องและ **เปิดอ่านได้จริงในตัวอ่าน** (FR-9)
- ไม่มีระบบกันก๊อปติดมาจากที่อื่น (Read24 ใส่การป้องกันเอง)
- ขนาดไม่เกิน 50MB, มีปก, มีสารบัญ
- ฟอนต์ไทย/การจัดหน้าแสดงผลถูกต้องบนจอเล็ก (≥360px, NFR-6)
- ผ่านการสแกนไวรัส

### 17.5 ข้อมูลสัญญา/สิทธิ์ที่ต้องเก็บ
ผูกกับสำนักพิมพ์แต่ละราย: ส่วนแบ่ง (70% ปกติ / 75–80% สำหรับเล่มที่ลงกับเราเจ้าเดียว ตาม PRD §9), เป็นเล่มเจ้าเดียวหรือไม่, เลขอ้างอิงสัญญา, ขายอย่างเดียว/ขาย+เช่า, พื้นที่ขาย, ช่วงเวลาสัญญา

### 17.6 ขั้นตอนสถานะหนังสือ
```
รับเข้า → ตรวจไฟล์ผ่าน → ตรวจคุณภาพ (เปิดอ่านจริง+ตั้งราคา+เช็กปก) → เผยแพร่
                                            └─ ไม่ผ่าน → ตีกลับ (แจ้งเหตุผล)
```
ตรงกับ `books.status`: ฉบับร่าง → เผยแพร่; ถ้ามีปัญหาภายหลัง → ระงับ (ไม่กระทบคนที่ซื้อไปแล้ว, FR-20)

### 17.7 จุดที่ต้องเช็กทางเทคนิค
- ทุกเล่มที่เผยแพร่ต้องมีไฟล์ EPUB อยู่จริงบน S3 (มีงานตรวจไฟล์หายก่อนเปิดตัว)
- ตั้งราคา/โหมดขายให้ครบ — เล่มที่เช่าไม่ได้ต้องไม่โชว์ปุ่ม "เช่า" (FR-4)
- เตรียมสคริปต์นำเข้าชุดเปิดตัว + ตรวจไฟล์อัตโนมัติ ก่อนเปิดให้ใช้จริง

> **คำถามที่ต้องเคาะ:** จำนวนเล่มเป้าหมายวันเปิดตัว + รายชื่อสำนักพิมพ์/ดีลเล่มเจ้าเดียวที่ปิดได้ทัน — กระทบทั้งจำนวนหนังสือและจุดขาย
